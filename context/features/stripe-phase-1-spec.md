# Stripe Phase 1 ‚Äî Core Infrastructure

## Overview

Lay the foundation for Stripe Subscriptions: schema additions, SDK singleton, plan-limit logic, billing DB helpers, and auth/session wiring so `isPro` flows through the JWT. **No user-visible UI and no webhook in this phase.** Goal is to land all the headless plumbing with full unit-test coverage so Phase 2 can wire Checkout, the webhook, and the UI on top.

Reference: [docs/stripe-integration-plan.md](docs/stripe-integration-plan.md) sections ¬ß1‚Äì3.1 (`src/lib/stripe.ts`, `src/lib/plan-limits.ts`, `src/lib/db/billing.ts`) and ¬ß3.2 (auth + types).

## Requirements

- Add two Stripe state columns to `User` so we can render renew/cancel UX in Phase 2 (`stripePriceId String?`, `stripeCurrentPeriodEnd DateTime?`). Use `prisma migrate dev --name add_stripe_subscription_state` (never `db push`).
- Install Stripe Node SDK: `npm install stripe`. The SDK ships its own types ‚Äî no `@types/stripe`.
- Create `src/lib/stripe.ts` ‚Äî Stripe singleton with `apiVersion` pinned, plus a `STRIPE_PRICE_IDS` map and `BillingInterval` type. Throws on missing `STRIPE_SECRET_KEY` so we fail fast at startup.
- Create `src/lib/plan-limits.ts` exposing `FREE_LIMITS = { items: 50, collections: 3 }`, `getUsage(userId)`, and `canCreate(userId, name)` returning `{ allowed, reason?, used, limit }`. Pro users always allowed.
- Create `src/lib/db/billing.ts` with `getOrCreateStripeCustomerId(userId, email, stripe)`, `setSubscriptionState(customerId, data)`, and `getBillingProfile(userId)`. All queries scoped by `userId` or `stripeCustomerId` ‚Äî never client-trusted IDs.
- Wire `isPro` through NextAuth: extend `src/auth.ts` JWT callback to always read `isPro` from DB (one extra `findUnique` per session validation). Mirror in `session()` so `session.user.isPro` is available in server components and API routes.
- Augment session/JWT types in `src/types/next-auth.d.ts` (add `isPro: boolean` to `Session.user` and `isPro?: boolean` to `JWT`).
- **Do not** add Pro gating to existing `createItem` / `createCollection` actions yet. That belongs in Phase 2.

## Unit Tests (required)

`plan-limits.test.ts` ‚Äî mock `prisma` from `src/lib/prisma`. Cover:

- Pro user ‚Üí `canCreate('items')` allowed regardless of count, `limit === Infinity`.
- Free user with `count < limit` ‚Üí allowed.
- Free user with `count === limit` ‚Üí not allowed, `reason === 'limit_reached'`.
- Free user with `count > limit` (just in case Pro grandfathered data exists) ‚Üí not allowed.
- Same coverage for `'collections'`.
- `getUsage` returns `{ items, collections, isPro }` with correct shape and `isPro` defaults to `false` when user not found.

`db/billing.test.ts` ‚Äî mock `prisma` and a fake Stripe object:

- `getOrCreateStripeCustomerId` returns existing `stripeCustomerId` without calling Stripe.
- When user has no `stripeCustomerId`, calls `stripe.customers.create({ email, metadata: { userId } })` and persists the returned `id` via `prisma.user.update`.
- `setSubscriptionState` writes all four fields (`stripeSubscriptionId`, `stripePriceId`, `stripeCurrentPeriodEnd`, `isPro`) keyed by `stripeCustomerId`.

Match the existing test conventions in `src/actions/items.test.ts` and `src/lib/db/items.test.ts` ‚Äî mock prisma client, no real DB hits.

## Key Gotchas

- **Migration command.** Use `prisma migrate dev --name add_stripe_subscription_state` ‚Äî never `prisma db push`. After running, commit the generated SQL file.
- **`isPro` is now on the JWT.** Existing code that does `await auth()` will start receiving `session.user.isPro`, but no consumer reads it yet ‚Äî that's fine.
- **Per-call DB read in JWT callback.** This is intentional. NextAuth's `trigger === 'update'` is unreliable for webhook-driven changes (see [context/research/stripe-integration-research.md](context/research/stripe-integration-research.md)). One `findUnique` with `select: { isPro: true }` against a Neon pooled connection is cheap.
- **Stripe SDK throws at import time** if `STRIPE_SECRET_KEY` is unset. That's deliberate ‚Äî fail loudly in dev so you remember to fill `.env`. Use a Stripe test key (`sk_test_‚Ä¶`) for now; real key wiring is Phase 2.
- **No webhook in this phase.** `setSubscriptionState` exists but has no caller until Phase 2. That's OK ‚Äî the unit test covers it.
- **Don't read `process.env.STRIPE_SECRET_KEY` in `plan-limits.ts` or `db/billing.ts`.** Only `src/lib/stripe.ts` should reach for env vars; everything else takes the `stripe` client as an argument so tests can inject a mock.
- **Existing 79 tests must still pass.** The auth changes are the highest-risk area; make sure the items/collections/auth-related tests still mock prisma in a way that satisfies the new `findUnique({ where: { id }, select: { isPro: true } })` call (mock returning `{ isPro: false }` is fine if anything explodes).

## Environment Variables

Already declared in `.env.example`. Phase 1 only needs:

```
STRIPE_SECRET_KEY=sk_test_...
```

The publishable key, webhook secret, and price IDs aren't read by Phase 1 code but should be left in `.env.example` for Phase 2.

## Testing (manual)

1. `npm run build` ‚Äî passes.
2. `npm test` ‚Äî all existing tests + new `plan-limits` and `db/billing` tests pass.
3. `npm run lint` ‚Äî clean.
4. Sign in to a dev account. In a browser devtool console run `await fetch('/api/auth/session').then(r=>r.json())`. Confirm `user.isPro === false`.
5. Manually flip `isPro = true` in the DB for that user. Reload. Confirm `user.isPro === true` in the session blob.
6. `npx prisma migrate status` ‚Äî clean.
